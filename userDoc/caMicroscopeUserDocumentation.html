Install and Config of Lite

A lite/local version of caMicroscope/quip is available, which functions similarly to the hosted (full) version, but does not require any of the below configuration. It adds functionality to load and save annotations to a json file locally using the same format as the hosted version, except with the file path as the image id. To see this, try drawing something and downloading the json data.

Install and Config of Hosted

Distribution of caMicroscope is designed as a set of microservice containers, and can be found at https://github.com/camicroscope/Distro/tree/refactor. This method requires docker and docker compose.

Configuration of services is done through environment variables, as passed through the docker compose file (caMicroscope.yml). The mounted volume, which is ‘images’ in the same directory as this repo by default, should be set consistently across all volumes in the compose file. Likewise, if exposed ports need to be changed, please change them in the compose file. The recommended configuration is to map the elevate container’s 4010 to whichever port is desired for users to access, and to expose no other ports in production, and to open what is needed only on a private network for debug or development purposes.

Services

Since the short names given for the services may not be clear, here is a description of each of the services in the compose file. Mongo is unmodified mongodb, which is used as a data provider for bindaas, which exposes preselected mongo functionality (see config/bindaas_projects) as REST API functions. IIP is a fastcgi module which translates openslide images to openseadragon compatible formats. Viewer hosts static files, and builds packages (see the “Extensions” section for more information on packages). Loader is used to extract thumbnails and slide metadata so that the viewer can understand the slide. Elevate is the security and routing container (see routes.json).

Recommendations

It is highly recommended to run the indexes in config/mongo_idx.js. Whether or not you wish to use validation is up to you, but the performance indexes at the bottom range from essential to extremely helpful.

By default, container logging has been disabled to prevent any possibility of data leak; this may be changed if desired, but use care with log files if you choose to use them. This does not change logging behavior within the containers themselves.

Further configuration may be accomplished by reviewing the docker compose documentation.

Running without Authentication

Each request, except to static files in viewer, must include a valid signed JWT. You can override this functionality, thus making all routes public by passing DISABLE_SEC=true to elevate. This choice means that the SECRET environment variable is unused, and the “Setting up a JWT Provider” section can be safely ignored. This is not recommended for production instances.

Setting up a JWT Provider

caMicroscope itself does not provide any user verification, and instead delegates that to other providers. This choice, however, means that those deploying caMicroscope will need to set up a JWT (json web token) provider. Mount a file resembling the given default login.html and set the path to your login file to REDIRECT in the compose file under the elevate service’s environment variables. In addition, be sure to set the public key or secret, as used for JWT verification, to SECRET in the compose file under the elevate service’s environment variables.

Google and auth0 can be used as JWT providers, among others.

Loading Images

The docker compose distribution includes an images folder. It is recommended that you put slides in this mounted directory if possible. Otherwise, you can use the web loader (./apps/loader/loader.html) to upload files. The web loader can also be used as an interface to get a thumbnail preview of any slide in the directory, as well as get metadata and post the slide to the database.

The loader container also has access to a batch slide loader, which uses a csv manifest, but this method is not recommended without further guidance at this point.

The Standard Viewer

The viewer can pan and zoom with either touch or mouse events, or be controlled using the zoom controls in the bottom right, above a navigator shows the position of the current view in context of the full image (Figure 2-1).

(Figure 2-1 - The standard Viewer)

A core functionality of caMicroscope is making and viewing annotations. Using the `Marks Tool`, you enter `Marks Tool`, and can change options using the `Marks Options`(Figure 2-2). Clicking `Marks Tool` twice changes the draw mode to `mutli`. An annotation is not saved until a name is provided and save is selected in single draw mode, or after each drawing is complete with a given name in multi draw mode.

(Figure 2-2 - under the drawing mode)

Annotations of the same group share a layer. Each layer can be turned on and off independently, and clicking on an annotation object should provide details, such as the internal id, the name, and any form responses associated with the particular annotation.(Figure 2-3)

(Figure 2-3 - layer manager and annotation detail)

The viewer has many tools and functions to assist with both individual and group operations. One of these is the share button, which provides a link to the image at the same position, zoom, and with the same visible layers as the current viewer, and can be used to bookmark or share specific observations in an image.

Depending on configuration, it can be useful to switch to a different viewer, such as a heatmap viewer. Many, but not all, tools are common between different viewer types.

Segmentations

What is segmentation?

The segmentation tool allows you to select a region of interest (ROI) and run a quick segmentation process on it. When it is finished, it displays a close button, a slider for adjusting the segmentation threshold and a current count of the segmented objects.

Extensions

caMicroscope is designed to be extended. Depending on the manner of extension, it may be a package or a new application. It is highly recommended to use a minimal methods of extension to avoid code duplication and issues with future upgrade compatibility.

Changing Package on default viewer

The viewer container builds package/package.js on build, and the resulting file is included in the viewer. This is used to add small custom functionality tweaks to the viewer.

Adding more applications alongside the default viewer

For more substantial modifications, it may make more sense to make a new application. These can be mounted somewhere (preferably somewhere in the apps folder) in the viewer service, and would include modification of the toolbar, or implementation of an entirely new functionality.

Contributing to CaMicroscope

caMicroscope is open source software. Any involvement and contribution with the caMicroscope project is greatly appreciated. Feel free to get directly involved in any of the repositories in the caMicroscope organization.

The caMicroscope project has persistent branches for development and release. It is highly recommended to make any changes off of the develop branch of a repository, and, when ready, create a PR to the develop branch of the source repository.

The creation of additional specializations, packages,  or applications should, in most cases, live as a separate repository.

For more information, or to propose ideas, please feel free to reach out to the active development team at ideas@camicroscope.org. Questions about the organization should be directed to info@camicroscope.org. (we need to make these emails, or similar…)
